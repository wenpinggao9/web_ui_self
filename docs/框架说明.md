# UI 自动化框架 — 设计说明

**技术栈**：自研编排 + LLM + Playwright  
**文档结构**：前半为**简版蓝图**（给领导/新人 5 分钟看懂）；后半为**详细设计**（问题、初衷、方案、关键规则）。

---

# 第一部分：简版蓝图

## 1. 项目定位

测试人员用**中文 Markdown** 写用例（像写 Word 一样），系统自动：

- 打开浏览器、按角色登录
- 把自然语言步骤翻译成浏览器操作
- 动态定位页面元素、执行并校验
- 输出 HTML 报告，并可选生成 Playwright 回归脚本

**测试不写代码、不写选择器**；AI 负责理解与判断，规则与缓存负责提速与稳定。

---

## 2. 要解决的核心问题（简表）

| 痛点 | 框架做法 |
|------|----------|
| 维护成本高 | 用例 = 自然语言；选择器执行期动态决策 |
| 改版易碎 | 五级定位链 + 加速库自愈；越跑越稳 |
| 业务耦合 | `业务知识.md` 插拔 API/角色；`core/` 不写死字段 |
| 假通过 | 后校验 LLM + 智能重试 |
| 难复盘 | 每步 DOM、提示词、LLM 原文、执行追踪落盘 |

## 6. LLM 用在哪（简表）

| 环节 | 频率 | 可否关闭 |
|------|------|----------|
| 动作规划 | 每 case 1 次 | 可用预规划动作注入 |
| 元素决策 L5 | L1~L4 未命中时 | 加速库命中则不调 |
| 就绪 / 后校验 | 按步骤 | `pre_readiness_check` / `post_step_check` |
| 语义断言 | 程序化失败兜底 | 可减少但不宜全关 |

# 第二部分：详细设计

> 以下各节统一结构：**要解决什么问题 → 设计初衷 → 实现方案 → 关键规则**。

---

## 8. 步骤①~③：用例进入系统

### 8.1 步骤① 解析器

| | |
|---|---|
| **问题** | 自然语言用例格式不统一，无法直接驱动浏览器 |
| **初衷** | 用固定 Markdown 模板降低写作成本，解析层只做结构化、不做语义理解 |
| **方案** | `core/parser/`：`#`~`###` → 模块路径；`#### 用例ID`；前置/步骤/预期列表 → `ParsedCase` |
| **规则** | 支持 `1.` / `-` 列表；资源定义走 ` ```yaml ` 块 |

### 8.2 步骤② 前置条件展开

| | |
|---|---|
| **问题** | 「已有收货地址」类状态无法直接点击，需变成可执行操作 |
| **初衷** | 状态型前置与操作型步骤分离；有 API 时优先接口造数，减少 UI 绕路 |
| **方案** | `PreconditionExpander`：LLM 或规则展开；`业务知识.md` 配置 API 前置 |
| **规则** | 登录类前置剔除（步骤④统一处理）；展开步骤插入 `步骤列表` 最前 |

### 8.3 步骤③ 用例排序

| | |
|---|---|
| **问题** | 多用例文件内存在隐式依赖（先建数据再消费） |
| **初衷** | 自动排执行顺序，减少人工标注依赖 |
| **方案** | 可选 `case_sort_llm`：LLM 推断边 → 拓扑排序 |
| **规则** | 默认关闭；单文件少量 case 可人工控制顺序 |

---

## 9. 步骤④⑤：会话与导航

### 9.1 步骤④ 登录与跨用例会话

| | |
|---|---|
| **问题** | 多角色、多用例重复登录耗时；多系统账号配置分散 |
| **初衷** | 角色与浏览器 context 一一对应；同批次内复用会话 |
| **方案** | `session/login.py` + `ProfileManager`；`role_contexts` 跨 case 传递 |
| **规则** | `cross_case_session: true` 时不关闭 context；**切回其他角色后回到原角色时 `reload` 同步数据**（`cross_case_session.py`）；tab 选择仍由 handoff 负责 |

### 9.2 步骤⑤ 模块导航

| | |
|---|---|
| **问题** | 深层菜单路径靠人工逐步点击易错 |
| **初衷** | 用例标题层级隐含导航路径，框架自动点菜单 |
| **方案** | `session/navigator.py` + `feature_selectors` / 菜单扫描 |
| **规则** | `auto_navigate: false` 时由用例步骤自行导航（如前审场景） |

---

## 10. 步骤⑥：动作规划（核心能力一）

| | |
|---|---|
| **问题** | 自然语言一步可能含多个操作；若规划层产出选择器，改版即全盘失效 |
| **初衷** | **两段式**：规划只产出「做什么」，执行期再决定「点哪里」 |
| **方案** | `ActionPlanner` + `prompts/action_plan.*.md`；输出 `PlannedAction{type, intent, value, role}` |
| **规则** | 每 case 约 1 次 LLM；复合句拆原子动作；剥离重复菜单点击；多角色按 `ExecutionBlock` 分块 |

**输入**：前置、步骤、预期、业务角色、当前 URL、技能知识  
**输出**：`已规划动作.json`（可审计、可预注入跳过 LLM）

---

## 11. 步骤⑧⑨：语义 DOM 与五级定位（核心能力二）

### 11.1 步骤⑧ 语义 DOM

| | |
|---|---|
| **问题** | 整页 HTML 过大无法喂给 LLM；ref/快照方案不利于跨次回归 |
| **初衷** | 抽取**可交互语义节点列表**（索引化），兼顾 LLM 与规则引擎 |
| **方案** | `v3_bridge` traverse → `semantic_dom.extract_items`；弹窗/表单/iframe 专项 |
| **规则** | L3/断言/后校验用**全量** items；仅 L5 受 `dom_limit` + `intent_window` 裁剪 |

**导航后 capture**：`operation_caused_navigation()` 为真时走 `wait_for_semantic_items_settle`（节点数连续稳定），写入 `settled: true`。**不按业务 URL 形态硬编码**。

### 11.2 步骤⑨ 五级定位链

| | |
|---|---|
| **问题** | 每次问 LLM 慢、贵、有波动；纯规则又覆盖不全 |
| **初衷** | **逐级降价**：能缓存/规则就不调模型；失败可降级、成功可沉淀 |
| **方案** | `LocatorResolver` 编排 L1~L5（见简图 §4） |

| 级别 | 模块 | 键维度 | 持久化 |
|------|------|--------|--------|
| L1 | `cache.py` | url + action + intent | `selector_cache/selector_cache.json` |
| L2 | `memory.py` | 同上 + 评分 | `selector_memory/selector_memory.json` |
| L3 | `intent_rule_engine` + skill | 组件类型 → build_* | 代码内规则 |
| L4 | `page_structure_learner` | route + DOM 指纹 + 模板 | `page_structure_learner/*.json` |
| L5 | `llm_decider` | DOM 索引 | 成功后回填 L1/L2/L4 |

**规则**：

- L1 含校验 + 三种自愈；失效则 evict
- 重试轮 `skip_acceleration` 仍走 L4，仅跳过 L1/L2 重复查
- 后校验失败：L1 evict、L2/L4 penalize

---

## 12. 步骤⑩~⑬：可靠性层

### 12.1 步骤⑩ 就绪检查

| | |
|---|---|
| **问题** | 弹窗遮挡、表单未填完就点提交 → 假点击 |
| **初衷** | 推进类操作前由 LLM 判断「现在能点吗」，并给出恢复动作 |
| **方案** | `readiness/pre_check.py`；`pre_readiness_check` 开关 |
| **规则** | 主要用于提交/保存/goto；恢复动作可先于主动作执行 |

### 12.2 步骤⑫⑬ 后校验与重试

| | |
|---|---|
| **问题** | 元素点到了但业务没变（假通过） |
| **初衷** | 执行后再问一次「效果对不对」；失败时换策略而非盲目重试 |
| **方案** | `post_check.py` + `retry.py`（`RetryController`） |
| **规则** | 最多 `post_step_max_retries`；改 value / 换选择器 / 排除已试；弹窗先规则关再就绪；`resolve_hint` 传入下轮定位 |

---

## 13. 步骤⑪：动作分发与 Tab 管理

| | |
|---|---|
| **问题** | 查看类操作新开 tab；提交后 tab 关闭；多角色多 context；后台 tab 数据过期 |
| **初衷** | **通用** tab/浏览器管理，不绑定「列表页/详情页」等业务概念 |
| **方案** | `PageSession` + `tab_follow` + `ActionDispatcher` |

| 机制 | 作用 |
|------|------|
| `list_anchor` | 记录列表 tab，多 tab handoff 锚点 |
| `pick_role_handoff_page` | case 间 / session 结束同步应停留的 tab |
| `_click_and_follow_navigation` | 新开 tab → `new_tab_opened` + `navigation_outcome` |
| `cross_case_session` reload | 跨角色切回时刷新当前页数据 |
| `invalidate_dom` | tab/URL 变化时清断言 DOM 缓存 |

---

## 14. 断言体系

| | |
|---|---|
| **问题** | 预期表述灵活（区域、语义、单选）；整页文本匹配易误报；DOM 半成品导致偶发失败 |
| **初衷** | 分层断言：快的先走，贵的兜底；已 settle 的 DOM 不重复重抓 |
| **方案** | `assert_scope.py` + `dispatcher._assert_text` |

```
FastAssert（同 URL settled 缓存）
  → 程序化（区域字面量 / 表格行 / radio 计数 / 否定）
  → [仅 unsettled 缓存] force_refresh 重抓
  → try_live_scoped_text（活页 XPath）
  → semantic_assert（LLM）
```

## 15. 步骤⑭⑮：输出与回归

| | |
|---|---|
| **问题** | AI 跑通一次后，发布门禁需要确定性脚本；失败需可复盘 |
| **初衷** | **AI 探索 + 脚本固化** 双模式；全链路落盘 |
| **方案** | `codegen.py` 生成 `playwright_*.py`；`ExecutionTrace` + `ObservabilityCollector` + `report.py` |

| 落盘文件 | 用途 |
|----------|------|
| `已规划动作.json` | 核对 LLM 是否理解用例 |
| `执行追踪.json` | 定位链、重试、page_switch |
| `可观测性.json` | LLM 调用链、五级命中率 |
| `使用的模型提示词.md` | 提示词版本追溯 |

## 16. 业务解耦

| | |
|---|---|
| **问题** | 不同系统登录、API、枚举差异大，写进框架则不可维护 |
| **初衷** | 框架只提供能力；业务只提供配置与用例 |
| **方案** | |

```
业务/<系统>/<项目>/
├── 业务知识.md      # API、角色、枚举、session 字段
├── 项目配置.yaml    # base_url、登录、账号
└── cases/*.md
```

`BusinessLoader.discover()` 从用例路径向上发现配置；`core/` 通过 `entity_discover` 等通用规则识别 URL 实体，**不写死业务字段名**。

## 18. 设计原则汇总

| # | 原则 | 说明 |
|---|------|------|
| 1 | 两段式 | 规划 = 意图；执行 = 定位 + 操作 |
| 2 | 降价链 | L1→L5，能不用 LLM 就不用 |
| 3 | 可沉淀 | 成功轨迹写入加速库，二次运行更快 |
| 4 | 通用会话 | Tab/handoff/reload 用机制信号，不按业务 URL 分支 |
| 5 | 通用 DOM | 导航/settle 用 `operation_caused_navigation`，不按页面类型分支 |
| 6 | 可审计 | 提示词、DOM、追踪、规划均可落盘 |
| 7 | 可回归 | 生成 Playwright 脚本供 CI 固化关键路径 |

## 20. 配置开关

`config.yaml` → `runner`：

| 开关 | 默认 | 影响 |
|------|------|------|
| `pre_readiness_check` | true | 步骤⑩ |
| `post_step_check` | true | 步骤⑫⑬ |
| `cross_case_session` | true | 跨用例浏览器复用 + 切回 reload |
| `auto_navigate` | false | 步骤⑤ |
| `case_sort_llm` | false | 步骤③ |

关闭 `post_step_check` 可降本提速，**假通过风险上升**，不建议用于发布门禁。
